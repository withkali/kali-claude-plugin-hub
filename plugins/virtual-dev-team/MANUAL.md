# 가상 개발팀 — 운영 매뉴얼 (Team Operating Manual)

이 문서는 `virtual-dev-team` 플러그인의 팀 헌장입니다. 슬래시 명령(`/new-service` 등)과 모든 에이전트가 이 규칙을 따릅니다.
(플러그인은 프로젝트 메모리를 자동 로드하지 않으므로, 이 매뉴얼의 핵심 규칙은 각 에이전트와 명령에 내장되어 있습니다. 사람을 위한 단일 참조본이 이 파일입니다.)

---

## 0. 역할 모델

| 주체 | 정체 | 책임 |
|---|---|---|
| **운영자(Operator)** | 사람 | 서비스 아이디어 인입, 최종 승인, 위험 작업 게이트 |
| **오케스트레이터** | 메인 Claude 세션 | 작업을 단계로 쪼개 적합한 에이전트·모델에 분배, 운영자에게 보고 |
| **파트 담당자(Agents)** | 플러그인 서브에이전트 | 자기 영역 작업, 질의/답변, 기록 |

오케스트레이터는 직접 큰 코딩을 하지 않고, 단계 → 적합 에이전트 + 모델로 위임하고 결과를 종합해 보고한다.

---

## 1. 모델 디스패치 정책 (작업 유형으로 결정)

`Agent` 호출 시 `model`을 항상 명시한다.

| 작업 유형 | 모델 | 예시 |
|---|---|---|
| 🔎 웹 검색·fetch·문서 조회 | **haiku** | API 문서, 버전 확인, 레퍼런스 수집 |
| 🛠 승인된 plan 실행 | **sonnet** | 코드 작성, 명령 실행, 설정 적용, 테스트 |
| 🧠 기획·설계·분석·트리아지·리뷰 | **opus** | 기획서, 아키텍처/스키마 설계, 장애 분석, 리뷰, 의사결정 |

동사 기준: `검색/조회/수집`→haiku, `구현/작성/실행/설정`→sonnet, `기획/설계/분석/판단/리뷰`→opus. 에이전트의 `model:`은 기본값이며 오케스트레이터가 단계에 맞게 오버라이드한다.

---

## 2. 단계별 워크플로우

### Phase 1 — 기획 정합 (활성)
1. 아이디어 인입 → `records/<product>/00-idea.md`
2. **planner**(opus) 기획서 초안 → `01-spec.md`
3. **designer** + 관련 **개발 파트**가 질의 작성 → `qa/`
4. **planner**(opus) 질의 반영해 기획서 보강(변경 이력)
5. 운영자 승인 → Phase 2

### Phase 2 — 개발 (활성)
1. 서버 필요 시 **infra-config**가 **사전 요구사항 질의서** 먼저 작성(`qa/`) → 답변 후 구성
2. 각 파트 개발. 설계는 opus → 승인 → sonnet 구현
3. 작업은 제품 repo PR, 진행 로그는 `worklog/`에 링크와 함께

### Phase 3 — 운영 (정의됨, 기본 비활성)
테스트·배포·모니터링·장애관리 에이전트는 정의돼 있으나 **기본 워크플로우에서 자동 호출하지 않는다.** `/activate-phase3`로 운영자가 활성화하면 편입한다.

---

## 3. 에이전트 디스패치 규칙

- 독립 작업은 병렬 호출, 의존 작업은 순차.
- 검색은 먼저 **researcher**(haiku)로 수집 후 설계/구현 에이전트에 전달.
- 호출 프롬프트엔 항상: (a) 무엇을/왜, (b) 관련 기획서·기록 경로, (c) 산출물 기록 위치, (d) 모델 의도.

---

## 4. 기록(Records) 프로토콜

모든 파트는 ① 자기 작업 ② 다른 파트 질의/답변 ③ 운영자 질의/답변 을 기록한다. 기록은 **작업 중인 프로젝트의 `records/`** 아래에 둔다.

```
records/
  README.md                 # 인덱스(제품·GitHub 링크 허브)
  _templates/               # 기획서/QA/작업로그/결정 템플릿
  <product-slug>/
    00-idea.md  01-spec.md  decisions.md
    qa/         worklog/
```

GitHub 매핑(인증 시): 제품 코드 = 배포 단위로 분리된 repo, 질의/답변 = Issues(파트 라벨), 작업 = PR, 중앙 기록 = `team-records` repo 인덱스. 로컬 파일은 "무엇을·어디서 확인"의 인덱스, 실제 산출물은 GitHub. 항상 링크. 상대 날짜는 절대 날짜로 기록.

---

## 5. 보안 가드레일 (절대 규칙)

1. 키·credential **하드코딩·커밋 금지.** env/시크릿 매니저로만 주입.
2. 저장소엔 `.env.example`(키 이름만)만. 실제 `.env`는 `.gitignore`.
3. Write/Edit는 플러그인의 **secret-scan 훅**이 자동 검사·차단. 우회 금지.
4. **클라이언트/웹에 키 하드코딩이 불가피한 경우:** 즉시 중단하고 운영자에게 (a) 어떤 키·왜, (b) 노출 위험, (c) 완화책(도메인/번들 제한, 백엔드 프록시, 키 회전·제한)을 보고하고 **명시 승인** 후에만 진행.
5. 배포 전 `/security-review`로 변경분 점검.

---

## 6. 자동화 / 자율성 규칙

기본 = 균형(로컬 전면 자동). 권한 allow/deny는 **플러그인이 아니라 프로젝트/유저 `settings.json`** 에 둔다(아래 §8).

**자율 진행(로컬·가역):** 파일 읽기/쓰기/수정, 디렉토리 생성, `git status/diff/log/add/commit/branch/checkout/stash`, 로컬 빌드·테스트·린트, dev 서버, 검색/fetch.

**운영자 승인 필요(비가역·공유·위험):** `git push`/force-push/`reset --hard`/`clean`/브랜치 삭제, 원격 repo 생성·삭제, PR/Issue 생성·머지, 모든 배포, prod DB 변경, 패키지 제거/다운그레이드, CI/CD 변경, `sudo`, 광범위 삭제, **클라/웹 시크릿 하드코딩**(§5.4).

막히면 우회하지 말고 근본 원인 수정 또는 운영자 보고. 한 번 승인 = 그 범위에 한함.

---

## 7. 새 제품 부트스트랩

1. slug(kebab-case) → `records/<slug>/` + `00-idea.md`
2. Phase 1 시작(planner → 질의 → 보강 → 승인) — `/new-service`로 자동화
3. 승인 후 repo 분리 결정 → (gh 인증 시) 승인받아 repo 생성 → `.gitignore`·`.env.example` 적용
4. `records/README.md` 인덱스에 제품·repo·Issue 링크 추가

---

## 8. 플러그인 구성 & 권한

- 플러그인이 제공: `agents/`(22), `hooks/`(secret-scan), `commands/`(`/new-service`, `/team-roster`, `/activate-phase3`).
- 플러그인이 **제공 못 함**: 권한 allow/deny 목록 → 이는 `settings.json`에 둔다. 균형 자동화 예시는 프로젝트 `.claude/settings.json`을 참고/복사하거나 `~/.claude/settings.json`에 두어 전역 적용한다.
